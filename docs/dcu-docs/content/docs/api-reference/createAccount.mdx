The `createAccount` endpoint is the first step for any new user. It establishes an on-chain identity that every group interaction references.

<Callout title="Standalone endpoint" type="info">
`createAccount` does not depend on a deployment's settings, so it is imported and called directly (not via `createDcuSdk`).
</Callout>

### Why Mint Two NFTs?
Under the Cardano **CIP-68 standard**, contracts cannot read data from tokens held in a wallet. `createAccount` mints a synchronized pair:
1. **User Auth (222) NFT** — sent to the user's wallet; their cryptographic "keycard".
2. **Reference (100) NFT** — locked at the `Account Validator`, holding the account datum (`display_name`, `contact`).

When joining a group, the contract checks the wallet for the auth token and reads the reference token's datum.

### What it does
- Mints the CIP-68 pair under the account policy.
- Locks the reference NFT at the Account Validator with the account datum.
- Sends the user auth NFT to the caller's wallet.
- Initialises `display_name` and `contact` as raw UTF-8 — both default to the wallet address when omitted.

## Configuration

```ts
import { OutRef } from "@lucid-evolution/lucid";

export type CreateAccountConfig = {
  /** Reference to a wallet UTxO spent for token-name entropy (CIP-68 derivation). */
  selected_out_ref: OutRef;

  /** Optional UTF-8 display name (username, ADA Handle, etc.). Defaults to address. */
  display_name?: string;

  /** Optional UTF-8 secondary contact identifier. Defaults to address. */
  contact?: string;
};
```

## Usage

```ts
import { createAccount } from "@tx-meta/dcu-kit";

const [utxo] = await lucid.wallet().getUtxos();

try {
  const txBuilder = await createAccount(lucid, {
    selected_out_ref: utxo,
    display_name: "alice",
    contact: "alice#1234",
  }).unsafeRun();

  const txHash = await txBuilder.sign.withWallet().complete().then((tx) => tx.submit());
  console.log("Account created! TxHash:", txHash);
} catch (error) {
  console.error("Failed to create account:", error);
}
```
