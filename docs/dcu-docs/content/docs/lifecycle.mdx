This page walks through a full Rotating Savings and Credit Association (ROSCA) lifecycle using the DCU Toolkit SDK, from identity creation to final exit.

<Callout title="Setup first" type="warn">
Group and treasury endpoints are bound to a deployment via `createDcuSdk(settingsPolicy)`, which requires the one-time settings setup. See [Getting Started](/docs/getting-started) before following this walkthrough.
</Callout>

<Callout title="Error handling omitted for brevity" type="info">
The examples below show the happy path without `try/catch`. In production, wrap every `unsafeRun()` call — or use `safeRun()`. See [Getting Started](/docs/getting-started) for the error-handling pattern.
</Callout>

## The Protocol Flow

```
createAccount → createGroup → joinGroup (×N) → startGroup
  → distributePayout (×N rounds) → [Pull: claimPayout] → exitGroup
```

`createAccount` is a standalone import; every group/treasury endpoint below comes from `createDcuSdk(settingsPolicy)`. Endpoint configs identify positions by their permanent CIP-68 **token suffix**, not by UTxO references.

```ts
import { createDcuSdk } from "@tx-meta/dcu-sdk";
const sdk = createDcuSdk(settingsPolicy);
```

---

## Step 1: Create an Account

Every participant establishes an on-chain identity before interacting with any group.

```ts
import { createAccount } from "@tx-meta/dcu-sdk";

// Any UTxO you own — used as entropy for CIP-68 token-name derivation.
const [utxo] = await lucid.wallet().getUtxos();

const txBuilder = await createAccount(lucid, {
  selected_out_ref: utxo,
  display_name: "alice",   // optional UTF-8 (defaults to wallet address)
  contact: "alice#1234",   // optional UTF-8 (defaults to wallet address)
}).unsafeRun();

const txHash = await txBuilder.sign.withWallet().complete().then((tx) => tx.submit());
```

Your wallet now holds a **User Auth (222)** NFT and the **Account Reference (100)** NFT is locked at the Account Validator. Keep the account's token suffix — it identifies you in later calls.

---

## Step 2: Create a Group

An admin initialises the ROSCA. See [createGroup](/docs/api-reference/createGroup) for the full `GroupDatum` (note: `num_rounds`, `start_time`, `member_count` must be `0`/`-1` at creation — `startGroup` sets them).

```ts
const txBuilder = await sdk.createGroup(lucid, {
  groupName: "Family Chama",
  groupDatum,        // full ruleset — see createGroup reference
  utxoToSpend,
}).unsafeRun();

const txHash = await txBuilder.sign.withWallet().complete().then((tx) => tx.submit());
```

---

## Step 3: Members Join

Each member calls `joinGroup` while the group is **not yet started**. Repeat for every participant up to `max_members`.

```ts
const txBuilder = await sdk.joinGroup(lucid, {
  groupTokenSuffix,     // from Step 2
  accountTokenSuffix,   // the member's account suffix from Step 1
}).unsafeRun();

const txHash = await txBuilder.sign.withWallet().complete().then((tx) => tx.submit());
```

`joinGroup` mints a treasury membership NFT, locks the collateral floor (`contribution_fee × collateral_rounds`), pays the `joining_fee` to the admin, and assigns the member the next sequential rotation slot.

---

## Step 4: Start the Group

The admin seals membership. This sets `is_started = true`, `num_rounds = member_count`, and anchors `start_time` to the transaction's lower bound. **Required before any payout** — and no further members can join afterward.

```ts
const txBuilder = await sdk.startGroup(lucid, { groupTokenSuffix }).unsafeRun();
const txHash = await txBuilder.sign.withWallet().complete().then((tx) => tx.submit());
```

---

## Step 5: Distribute Each Round

Once a round's time gate opens, anyone can submit `distributePayout` — it is **permissionless** and always pays the round's borrower (the member at the current slot), never the submitter. Call it once per round, sequentially.

```ts
const txBuilder = await sdk.distributePayout(lucid, { groupTokenSuffix }).unsafeRun();
const txHash = await txBuilder.sign.withWallet().complete().then((tx) => tx.submit());
```

The pot is pro-rata: `contribution_fee × active_members`. Under **Push** mode it goes straight to the borrower's wallet; under **Pull** it is earmarked into the borrower's own treasury for later claim (Step 6).

---

## Step 6: Claim a Pull-mode Payout

In a **Pull**-mode group, distribute earmarks the pot into the borrower's treasury (`claimable_balance`) rather than paying a wallet. The borrower withdraws it whenever they like — to any address — by proving possession of their member token.

```ts
const txBuilder = await sdk.claimPayout(lucid, {
  accountTokenSuffix,
  destinationAddress,   // optional — claim to a fresh address (lost-wallet recovery)
}).unsafeRun();

const txHash = await txBuilder.sign.withWallet().complete().then((tx) => tx.submit());
```

Push-mode groups skip this step — the borrower is paid directly in Step 5.

---

## Step 7: Exit or Terminate

After the group matures (`start_time + num_rounds × interval_length`), members exit and reclaim their remaining collateral; any unclaimed Pull earmark is returned automatically via the burn path.

```ts
const txBuilder = await sdk.exitGroup(lucid, { groupTokenSuffix }).unsafeRun();
const txHash = await txBuilder.sign.withWallet().complete().then((tx) => tx.submit());
```

For a member who exited early into a `PenaltyState`, the admin uses [`terminateGroup`](/docs/api-reference/terminateGroup) to claim the penalised position and recover the locked funds.

---

## Summary

| Step | Endpoint | Who calls it |
|------|----------|--------------|
| 1 | `createAccount` | Every participant |
| 2 | `createGroup` | Admin |
| 3 | `joinGroup` | Each member (×N) |
| 4 | `startGroup` | Admin (seals membership) |
| 5 | `distributePayout` | Anyone (permissionless; pays the borrower) |
| 6 | `claimPayout` | Borrower (Pull mode only) |
| 7 | `exitGroup` | Member leaving |
| - | `terminateGroup` | Admin (early-exit cleanup) |
